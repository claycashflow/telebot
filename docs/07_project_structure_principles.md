# 프로젝트 구조 설계 원칙

## 1. 문서 목적
이 문서는 `docs` 아래 정의된 도메인, PRD, 사용자 시나리오, 로직, 프롬프트, 데이터 스키마를 바탕으로 프로젝트 구조 설계의 공통 원칙을 정의한다.

목적은 특정 프레임워크나 구현체보다 먼저, 이 프로젝트가 어떤 구조와 경계를 가져야 일관성 있게 확장 가능한지를 정하는 것이다.

## 2. 모든 스택에 공통인 최상위 원칙
### 2.1 도메인 우선
- 프로젝트 구조는 기술 스택보다 도메인 흐름을 먼저 반영해야 한다.
- 핵심 흐름은 `입력 -> 검증 -> 판정 -> 리포트 생성 -> 전달` 순서를 기준으로 나눈다.
- 시장 저점 판정 규칙은 이 프로젝트의 중심 도메인이므로 가장 명확한 경계로 분리한다.

### 2.2 규칙 기반 판정 우선
- 시스템은 임의 추론보다 문서화된 규칙 기반 판정을 우선해야 한다.
- 리포트, 봇 응답, 보조 로직은 모두 판정 결과를 소비하는 계층이어야 하며 판정 규칙을 우회하면 안 된다.

### 2.3 수동 입력 경로 보존
- 자동 데이터 수집이 추가되더라도 수동 입력 경로는 독립적으로 유지해야 한다.
- 데이터 수집 방식은 달라도, 검증 이후의 처리 흐름은 동일해야 한다.

### 2.4 입력과 출력의 명시성
- 입력 스키마와 출력 형식은 코드 전역에 흩어지지 않도록 명시적 계약으로 관리해야 한다.
- 입력 구조는 `06_data_schema.md`, 출력 구조는 `05_prompt_spec.md`를 기준으로 삼는다.

### 2.5 구현보다 문서 정합성 우선
- 로직, 스키마, 리포트 형식이 바뀌면 구현보다 먼저 문서와의 정합성을 점검해야 한다.
- 구조 설계는 문서 변경 영향을 좁은 범위에 가두는 방향이어야 한다.

## 3. 의존성 및 레이어 원칙
### 3.1 권장 레이어
- `interfaces`: CLI, 텔레그램, 배치 실행, 수동 입력 등 외부 진입점
- `application`: 유스케이스 조합, 흐름 제어, 에러 처리, 실행 단위 orchestration
- `domain`: 판정 규칙, 점수 계산, 상태 모델, 리포트 생성 규칙
- `infrastructure`: 텔레그램 API, 환경변수 로딩, 파일 입출력, 외부 데이터 수집

### 3.2 의존 방향
- `interfaces -> application -> domain`
- `infrastructure -> application` 또는 `infrastructure -> domain contract 구현`
- `domain`은 어떤 외부 API, 프레임워크, 전송 방식에도 의존하지 않는다.

### 3.3 도메인 규칙의 독립성
- 저점 판정 로직은 텔레그램 전송 코드와 같은 계층에 두지 않는다.
- 리포트 생성도 가능하면 도메인 또는 응용 계층의 순수 로직으로 유지하고, 메신저 포맷팅만 인터페이스 계층에서 담당한다.

### 3.4 계약 우선 연결
- 레이어 간 연결은 구체 구현보다 계약과 데이터 모델을 통해 이뤄져야 한다.
- 예를 들어 `InputData`, `JudgementResult`, `ReportPayload` 같은 명시적 구조를 경계 타입으로 사용한다.

### 3.5 교체 가능성
- 자동 수집기, 메신저 전송기, 입력 방식은 교체 가능해야 한다.
- 수집기나 전송기가 바뀌어도 `application`과 `domain`의 핵심 흐름은 변하지 않아야 한다.

## 4. 코드 및 네이밍 원칙
### 4.1 디렉토리 네이밍
- 디렉토리는 기술명보다 역할명으로 짓는다.
- 예시: `domain`, `application`, `interfaces`, `infrastructure`, `tests`, `config`
- `utils` 같은 포괄 디렉토리는 최소화하고, 생기더라도 범위를 좁게 제한한다.

### 4.2 파일 네이밍
- 파일명은 책임이 드러나야 한다.
- 예시: `market_input.py`, `bottom_judgement_service.py`, `report_formatter.py`, `telegram_sender.py`
- `final.py`, `temp.py`, `helper.py`처럼 책임이 불분명한 이름은 피한다.

### 4.3 코드 네이밍
- 도메인 용어는 문서 용어와 일치시킨다.
- `bottom_pattern`, `disparity_20`, `vkospi`, `semiconductor_earnings_view` 같은 필드는 문서와 동일한 이름을 우선 사용한다.
- 상태값과 enum은 문서의 허용값과 일치해야 한다.

### 4.4 함수와 클래스 책임
- 함수는 한 단계의 책임만 가져야 한다.
- 검증, 판정, 리포트 생성, 전송을 한 함수에 섞지 않는다.
- 클래스가 필요하다면 상태 보관보다 역할 캡슐화가 분명할 때만 도입한다.

### 4.5 코드 스타일
- 입출력 처리와 순수 계산 로직을 분리한다.
- 숫자 상수는 코드에 흩뿌리지 말고, 규칙 이름 또는 설정 가능한 상수로 모은다.
- 투자 자문처럼 보일 수 있는 표현 생성 로직은 프롬프트/리포트 계층에 한정한다.

## 5. 테스트 및 품질 원칙
### 5.1 도메인 로직 우선 테스트
- 가장 먼저 테스트해야 할 대상은 저점 판정 로직이다.
- 점수 계산, 구간 판정, 허용값 검증, 리포트 구성 규칙은 단위 테스트로 고정한다.

### 5.2 경계값 테스트
- 하락률, 이격도, 유가, GDP 등 임계값이 있는 지표는 경계값 테스트를 반드시 포함한다.
- 예시: 10, 18, 23, 80, 92, 88, 100, 4.0 부근

### 5.3 입력 검증 테스트
- 필수 필드 누락, 타입 오류, 허용되지 않은 enum 값, 음수/양수 방향 오류를 검증해야 한다.
- 수동 입력과 자동 수집 입력이 같은 검증 로직을 통과하는지도 확인해야 한다.

### 5.4 리포트 품질 테스트
- 리포트는 최소한 필수 섹션이 빠지지 않는지 확인해야 한다.
- 숫자 데이터와 해석 의견이 분리되는지, 과도한 확신 표현이 없는지도 점검 대상이다.

### 5.5 통합 테스트 범위
- `입력 -> 검증 -> 판정 -> 리포트 -> 전송 요청 생성`까지는 통합 테스트로 검증한다.
- 외부 텔레그램 API 호출은 실제 네트워크 대신 모킹 또는 계약 테스트로 검증한다.

## 6. 설정, 보안, 운영 원칙
### 6.1 설정 분리
- 토큰, 채널 ID, 외부 API 설정, 실행 모드는 코드에 하드코딩하지 않는다.
- 민감 정보와 환경별 설정은 `.env` 또는 동등한 설정 계층으로 분리한다.

### 6.2 비밀값 보호
- 텔레그램 토큰, 외부 API 키, 운영 채널 정보는 로그나 예외 메시지에 노출하지 않는다.
- 예제 파일과 실제 비밀값 파일을 분리한다.

### 6.3 운영 모드 구분
- 최소한 `local`, `test`, `prod` 수준의 실행 문맥을 구분할 수 있어야 한다.
- 운영 모드에 따라 로그 수준, 전송 대상, 외부 호출 여부를 통제할 수 있어야 한다.

### 6.4 실패 대응
- 자동 수집 실패, 입력 검증 실패, 전송 실패는 각각 분리된 오류로 다뤄야 한다.
- 실패가 발생해도 원인 추적이 가능하도록 로그와 에러 메시지를 구조화해야 한다.
- 외부 전송 실패가 판정 로직 자체를 손상시키지 않도록 분리해야 한다.

### 6.5 운영 관찰성
- 최소한 실행 시각, 입력 출처, 판정 결과, 전송 성공 여부는 남길 수 있어야 한다.
- 다만 로그에는 민감 정보나 불필요한 원시 데이터 전체를 남기지 않는다.
