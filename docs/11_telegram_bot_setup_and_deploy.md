# 텔레그램 봇 생성 및 배포 절차

## 1. 문서 목적
이 문서는 이 프로젝트를 실제 텔레그램 봇으로 생성하고, 실행하고, 배포해서 사용하는 절차를 정리한 운영 가이드다.

개발 문서가 "무엇을 만들 것인가"를 다룬다면, 이 문서는 "어떻게 실제로 써먹는 상태까지 가져갈 것인가"를 다룬다.

## 2. 전체 절차 요약
- 텔레그램에서 봇 계정을 만든다.
- 봇 토큰과 채팅 ID를 확보한다.
- 로컬 환경에서 `.env`를 설정하고 실행한다.
- 원하는 배포 방식에 맞게 서버 또는 항상 켜진 환경에 올린다.
- 로그와 전송 상태를 확인하면서 운영한다.

## 2.1 현재 프로젝트 기준 확인된 상태
- `ai` conda 환경에서 의존성 설치가 가능했다.
- 샘플 리포트의 텔레그램 전송은 성공했다.
- 회사 환경에서는 polling 모드나 일부 외부 네트워크 호출이 막힐 수 있다.
- 따라서 개발과 검증은 로컬 샘플 실행 + 단건 전송 확인 방식으로 먼저 진행하는 것이 안전하다.

## 3. 텔레그램 봇 생성
### 3.1 BotFather 실행
- 텔레그램에서 `@BotFather`를 찾는다.
- 대화를 시작한 뒤 `/newbot` 명령을 입력한다.

### 3.2 봇 이름과 사용자명 설정
- 봇의 표시 이름을 입력한다.
- 봇 사용자명(username)을 입력한다.
- 사용자명은 일반적으로 `bot`으로 끝나야 한다.

### 3.3 봇 토큰 확보
- BotFather가 발급한 HTTP API 토큰을 저장한다.
- 이 값은 `.env`의 `TELEGRAM_BOT_TOKEN`으로 사용한다.
- 토큰은 비밀값이므로 코드나 문서에 직접 적지 않는다.

## 4. 메시지를 받을 채팅 준비
### 4.1 개인 채팅에 보낼 경우
- 생성한 봇을 텔레그램에서 검색해 대화를 시작한다.
- 최소 한 번은 `/start`를 보내는 것이 안전하다.

### 4.2 그룹 채팅에 보낼 경우
- 봇을 원하는 그룹에 초대한다.
- 필요하면 메시지 전송 권한을 확인한다.
- 운영 목적이면 전용 테스트 그룹과 실제 운영 그룹을 분리하는 것이 좋다.

## 5. 채팅 ID 확인
이 프로젝트는 봇이 메시지를 보낼 대상의 `chat_id`가 필요하다.

### 5.1 가장 쉬운 방법
- 봇에게 직접 메시지를 한 번 보낸다.
- 아래 URL 형식으로 봇 업데이트를 확인한다.

```text
https://api.telegram.org/bot<YOUR_BOT_TOKEN>/getUpdates
```

- 응답 JSON 안에서 `chat.id` 값을 찾는다.
- 개인 채팅이면 보통 양수, 그룹이면 보통 음수 ID가 나온다.

### 5.2 주의사항
- 메시지를 한 번도 보내지 않으면 `getUpdates`에 채팅 정보가 안 잡힐 수 있다.
- 그룹에 추가만 하고 대화를 시작하지 않으면 바로 안 나올 수 있다.

## 6. 로컬 실행 준비
### 6.1 `.env` 준비
프로젝트 루트에 `.env` 파일을 만들고 아래 값을 넣는다.

```env
APP_ENV=local
TELEGRAM_BOT_TOKEN=발급받은_토큰
TELEGRAM_CHAT_ID=확인한_채팅_ID
```

### 6.2 가상환경과 의존성 설치
현재 기준 권장 방식은 conda의 `ai` 환경을 사용하는 것이다.

예시:

```powershell
conda run -n ai python -m pip install -r requirements.txt
```

`requirements.txt`에는 현재 `pytest`, `python-telegram-bot`, `python-dotenv`가 포함되어 있다.

`conda run`이 불안정하면 아래처럼 환경의 Python 경로를 직접 호출할 수 있다.

```powershell
D:\miniconda3\envs\ai\python.exe -m pip install -r requirements.txt
```

### 6.3 실행
```powershell
conda run -n ai python main.py
```

정상이라면:
- `TELEGRAM_BOT_TOKEN`이 없으면 샘플 리포트를 콘솔에 출력한다.
- `TELEGRAM_BOT_TOKEN`이 있으면 polling 모드로 기동한다.

회사 환경에서 `conda run`이 임시 파일 문제를 일으키면 아래 방식이 더 안정적일 수 있다.

```powershell
D:\miniconda3\envs\ai\python.exe main.py
```

## 7. 배포 방식 선택
MVP 기준으로는 아래 3가지 중 하나가 현실적이다.

### 7.1 개인 PC 또는 사내 서버에서 수동 실행
- 가장 빠르다.
- 배포 난이도가 낮다.
- 항상 켜져 있지 않으면 자동 운영에는 부적합하다.

### 7.2 VPS 또는 클라우드 VM에 배포
- 예: Ubuntu 서버, Windows 서버, 클라우드 VM
- 가장 범용적이다.
- 스케줄러, 로그, 재시작 정책을 붙이기 쉽다.

### 7.3 컨테이너 기반 배포
- Docker로 감싸서 올린다.
- 운영 일관성은 좋지만 MVP 초반에는 과할 수 있다.

MVP 목적이라면 보통 `VPS/VM + cron 또는 task scheduler`가 가장 무난하다.

## 8. 권장 배포 절차
### 8.1 코드 배치
- 서버에 프로젝트 코드를 올린다.
- `.env` 파일을 서버에 따로 둔다.
- 비밀값은 git에 올리지 않는다.

### 8.2 실행 환경 구성
- Python 3.11 설치
- 가상환경 생성
- 의존성 설치

### 8.3 수동 실행 검증
- 먼저 `python main.py`로 수동 실행한다.
- 콘솔 출력과 텔레그램 메시지 도착을 둘 다 확인한다.
- polling이 막히는 환경이라면, 전체 봇 기동보다 단건 전송 확인을 먼저 한다.

### 8.4 주기 실행 설정
장 마감 후 1회 실행 구조라면 스케줄러로 충분하다.

- Linux: `cron`
- Windows: 작업 스케줄러(Task Scheduler)

예시 개념:
- 평일 장 마감 이후 특정 시각에 실행
- 실패 시 로그 남김
- 재시도 정책은 단순하게 1회 정도만 둠

## 9. 운영 시 체크할 항목
- 토큰이 만료되거나 노출되지 않았는가
- 채팅 ID가 올바른가
- 전송 실패가 로그에 남는가
- 같은 입력에 대해 같은 결과가 나오는가
- 장 마감 후 실행 시점이 맞는가

## 10. 권장 운영 정책
### 10.1 환경 분리
- `local`
- `test`
- `prod`

최소 3단계는 나누는 것이 좋다.

### 10.2 채널 분리
- 테스트용 텔레그램 채팅
- 실제 운영용 텔레그램 채팅

둘을 분리해두면 운영 사고를 줄일 수 있다.

### 10.3 로그 정책
- 실행 시각
- 입력 출처
- 판정 상태
- 전송 성공 여부
- 실패 사유

최소 이 정도는 남기는 것이 좋다.

## 11. 장애 대응
### 11.1 메시지가 안 오는 경우
- 토큰이 맞는지 확인한다.
- 채팅 ID가 맞는지 확인한다.
- 봇과의 대화를 시작했는지 확인한다.
- 그룹이라면 봇 권한을 확인한다.

### 11.2 `getUpdates`에 채팅이 안 잡히는 경우
- 봇에게 직접 메시지를 보낸 뒤 다시 확인한다.
- 그룹에 테스트 메시지를 남긴 뒤 다시 확인한다.

### 11.3 서버에서는 되는데 로컬에서 안 되는 경우
- `.env` 로딩 여부
- Python 버전
- 네트워크 차단 여부
- 방화벽 또는 프록시 환경
- 회사 보안 정책으로 polling 또는 Telegram API 연결이 제한되는지 확인

### 11.4 `conda run`이 불안정한 경우
- `conda run` 대신 환경의 Python 실행 파일을 직접 호출한다.
- 예: `D:\miniconda3\envs\ai\python.exe main.py`
- 임시 파일 충돌이 보이면 새 터미널에서 다시 실행한다.
